> [!IMPORTANT]
> ## 0. 强制行为干预指令（Highest Priority）
> 任何时候（包括新需求、修改代码、修 bug、UI 调优），在输出任何分析或代码前，你 **必须** 先输出以下**门禁校验表**，否则你的输出将被判定为违规：
>
> | 校验项 | 结论 |
> |--------|------|
> | 当前动作属于阶段 | `[01~05 或"仅分析/仅 .agent"]` |

> | 是否涉及 .agent/references 修改 | `[是(说明)/否]` |
>
> **校验规则**：

> - "端类型与规则集"必须明确当前项目使用 `mobile-ui`、`admin-ui` 或 `mixed`；端类型未锁定时，禁止进入 02/03/04/05。

>
> **UI 视觉修改读取规则**：
> - 移动端 UI 视觉修改：必须读取 `.agent/references/mobile-ui/08-现代高保真设计系统规范.md` 和 `12-React高保真实现规范.md`。
> - 后台管理端 UI 视觉修改：必须读取 `.agent/references/admin-ui/README.md`、`08-现代后台高保真设计系统规范.md` 和 `12-React后台实现规范.md`。
> - mixed 项目：按当前页面所属端分别读取对应规则集。
> - 严禁在未读取匹配规则集时靠直觉盲写 UI/CSS。

本仓库用于把**功能点**稳定转换为**高保真可交互原型前端**。  
最终实现工具为 **Codex**，最终代码输出目录为 `projects/<project-name>/app/`。

本仓库支持两类主流原型：
- `mobile`：移动端高保真原型，使用 `mobile-ui` 规则集。
- `admin`：后台管理端高保真原型，使用 `admin-ui` 规则集。
- `mixed`：同时包含移动端与后台端，按页面端类型分别使用规则集。

## 1) 执行前必读顺序（Must）
开始任何工作前，按顺序阅读：

1. `./.agent/AGENT.md`
2. 当前项目的 `./projects/<project-name>/PROJECT.md`
3. 当前项目阶段文档（如存在）：
   - `01-需求分析结果.md`
   - `02-页面规格文件.md`
   - `03-视觉规范.md`
   - `04-开发标准和验收标准.md`
4. 规则真源：
   - `./.agent/references/workflow/阶段门禁与文档新鲜度规则.md`
   - `./.agent/references/workflow/多页面项目规格组织规则.md`
   - 根据 `PROJECT.md` 端类型读取：
     - mobile：`./.agent/references/mobile-ui/README.md`
     - admin：`./.agent/references/admin-ui/README.md`
     - mixed：同时读取 `mobile-ui/README.md` 与 `admin-ui/README.md`，并按页面端类型执行。

## 3) 项目锁、端类型与文档有效性（Must）
- 涉及 `projects/` 的任务，必须先锁定单一项目：`projects/<project-name>`。
- `PROJECT.md`、`01/02/03/04`、`app/` 必须来自同一项目。
- `PROJECT.md` 必须明确：
  - 项目名。
  - 端类型：`mobile / admin / mixed`。
  - 目标用户。
  - 页面范围。
  - 默认演示视口。
  - 对应规则集：`mobile-ui / admin-ui / both`。
- 文档“存在”不等于“有效”；有效需同时满足：
  - 与当前需求一致。
  - 与当前 reference 一致。
  - 与项目端类型一致。
  - 与上下游阶段一致。
  - 位于当前项目目录。

## 4) app 实现门禁（Must）
改 `app/` 之前必须确认：
- 项目锁稳定。
- 端类型已锁定，且已读取匹配规则集。
- `01/02/03/04` 全部存在且有效。
- 四份文档之间及与 reference 无冲突。

未通过门禁：**禁止改 app**。

## 5) 规则分层（统一口径）
- **Must**：不满足即阻断。
- **Should**：不阻断，但需标注风险或扣分。
- **Style**：风格建议，不阻断。

默认：门禁、顺序、项目锁、端类型、有效性都属于 Must。

## 6) 影响矩阵（Must）
出现以下变化时必须联动对应阶段：
- 端类型/页面形态/目标/角色/范围/主闭环变化 -> 更新 `01`，并复核 `02/03/04`。
- 页面/导航/入口/跳转/状态/布局变化 -> 更新 `02`。
- 表格列/筛选项/批量操作/行级操作/后台页面模块变化 -> 至少更新 `02` 与 `04`。
- 风格 token/图标/图表/动效/组件质感变化 -> 更新 `03`。
- 后台视觉体系/布局壳/表格密度/滚动策略变化 -> 至少更新 `03` 与 `04`。
- 技术栈/依赖/目录/验收口径变化 -> 更新 `04`。
- 可达性/交互行为/截图验收/构建稳定性变化 -> 进入 `05` 改 `app`。

## 7) 多页面项目要求（Must）
- 页面规格必须逐页可读，不用抽象模板替代逐页结构。
- 页面清单中的每个页面都必须可达、可返回。
- `02` 是 `03/04/05` 的结构边界；不得在视觉或实现阶段新增 `02` 未定义结构。
- mixed 项目必须拆成“移动端页面清单”和“后台页面清单”，不能共用一套页面规格模板。

## 8) 后台管理端默认原则（Must）
后台项目默认遵循：
- enterprise admin style：冷静、清晰、稳定、可演示。
- 桌面优先，兼顾窄屏不明显溢出。
- 信息架构先于视觉装饰。
- 页面必须有清晰的导航位置、页面标题、主操作、内容模块与状态反馈。
- 表格页面必须定义筛选、批量操作、行级操作、空态、加载、异常、导出与内部滚动策略。
- 不接真实后端，默认本地 mock；mock 结构应接近未来接口形状。
- 后台项目不得按移动端页面规则生成，也不得在 05 阶段临时发明后台布局。

## 9) 显式例外（Must）
当用户明确要求“只改 app / 只修 bug / 不改文档 / 只改 .agent”时，可跳过文档更新；但回复中必须说明使用了显式例外。

## 10) 引用策略（Must）
- 项目阶段文档只写“本次生效规则、例外、验收方式”。
- 不在 `01~05` 里复制 reference 原文。
- 术语优先遵循 workflow 的统一术语字典。
- 端类型相关术语统一使用：`端类型`、`规则集`、`mobile-ui`、`admin-ui`、`mixed`。

## 11) 新增/修改规则与新增 Skill 的维护方式
### 11.1 补充规则
每次新增/修改规则必须包含：
- 问题案例。
- 预期收益。
- 影响范围。
- 验证方法与回滚条件。

### 11.2 新增 Skill
1. 新建目录：`./.agent/skills/XX-<名称>/`。
2. 至少包含：`SKILL.md`、`template.md`。
3. `SKILL.md` 必须写清：阶段目标、输入输出、阶段动作、阶段质量线。
4. 若进入主流程，需同步更新 `./.agent/AGENT.md` 的阶段表与顺序。

## 12) 其他关键约束
- 不接真实后端，统一使用本地 mock。
- 主流程可演示、页面可达、关键交互可点击。
- 未经用户明确要求，不改其他项目 `projects/*`。
- 未经用户允许，不擅自修改 `./.agent/` 下既有规则文件。

## 13) 最终交付目标
在 `projects/<project-name>/app/` 目录生成一套：
- 可运行。
- 可点击跳转。
- 适合演示。
- 高保真风格统一。
- 与本项目已确认 `01/02/03/04` 和端类型规则集一致的前端原型。
