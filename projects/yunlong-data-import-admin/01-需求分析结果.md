# 需求分析结果

## 1. 项目目标
- 项目名称：yunlong-data-import-admin
- 端类型：admin
- 对应规则集：admin-ui
- 默认演示视口：1366x768；兼顾 1440x900
- 原型用途：给团队演示一个学校数据表格批量导入平台的后台管理端流程。
- 主要演示对象：项目团队成员、检测机构业务人员、教育局侧运营人员。
- 本次输出目标：锁定后台端原型目标、业务范围、核心角色、主流程和页面优先级，为 02 页面规格文件提供输入。

## 2. 业务问题与设计目标
### 2.1 当前要解决的问题
- 检测机构会按学校交付学生数据表格，人工逐个处理文件效率低。
- 文件中包含身份证、年级、班级、姓名等关键字段，导入时需要看到每个文件的处理状态。
- 导入过程可能出现字段缺失、身份证异常、班级异常、姓名缺失等问题，需要将问题数据导出给人工修正。
- 团队需要一个可演示的后台原型，说明批量导入、进度跟踪、异常导出和历史查看的完整流程。

### 2.2 本次原型希望体现的价值
- 让演示对象一眼理解这是一个标准后台管理系统，而不是移动端或看板页。
- 展示从文件选择、批量导入、状态进度、异常导出到历史查看的主流程。
- 用 mock 数据覆盖待导入、成功、异常等关键状态。
- 为后续页面规格、视觉规范和高保真实现提供稳定边界。

## 3. 核心角色
### 3.1 必须进入原型的角色
- 角色名称：导入运营人员
  - 核心职责：接收检测机构提供的学校表格文件，发起批量导入，查看导入状态并导出问题数据。
  - 在原型中的主要操作：选择文件、全部开始、单文件开始导入、搜索/筛选队列、导出问题数据、删除文件、查看导入历史。

- 角色名称：团队演示观看者
  - 核心职责：理解该平台如何支撑学校数据批量导入与异常处理。
  - 在原型中的主要操作：跟随演示查看导入队列、状态变化、问题导出和历史结果。

### 3.2 可弱化或暂不进入原型的角色
- 角色名称：系统管理员
  - 原因：本次不做权限、字段模板、系统配置和用户管理。

- 角色名称：学校老师/学校管理员
  - 原因：本次只演示检测机构或教育局侧后台导入流程，不做学校端修正流程。

## 4. 本次原型范围
### 4.1 纳入本次原型的模块
- 模块名称：后台全局壳
  - 所属端：admin
  - 目标：提供左侧导航、顶部栏、面包屑和内容区，让页面符合后台管理端结构。
  - 演示价值：验证 workspace 的 admin-ui 规则是否能生成标准后台页面，而不是移动端布局。

- 模块名称：表格导入页
  - 所属端：admin
  - 目标：展示文件选择、统计卡片、文件导入队列、批量导入和异常导出。
  - 演示价值：承载本项目主流程，是 P0 页面。

- 模块名称：导入历史页
  - 所属端：admin
  - 目标：展示状态不是“待导入”的文件列表，并包含操作时间。
  - 演示价值：体现导入完成后的结果沉淀与复查入口。

- 模块名称：mock 导入流程
  - 所属端：admin
  - 目标：用本地数据模拟文件导入进度、成功、异常、问题数据 CSV 导出。
  - 演示价值：让原型无需后端也能完成可点击演示。

### 4.2 不纳入本次原型的内容
- 内容：真实后端 API
  - 原因：当前阶段是高保真原型验证，默认 mock first。

- 内容：真实 Excel 解析
  - 原因：演示重点是后台流程和状态，不是文件解析能力。

- 内容：字段自动匹配独立模块
  - 原因：用户明确只需要导入队列与状态，不单独展示字段匹配页面。

- 内容：问题数据列表独立模块
  - 原因：问题数据通过导出 CSV 体现，不单独占用页面模块。

- 内容：导入日志独立模块
  - 原因：本次范围只保留表格导入和导入历史。

- 内容：复杂多级菜单、权限配置、系统管理
  - 原因：会稀释主流程，不利于测试 admin-ui 规则是否稳定生成目标页面。

## 5. 核心业务闭环
> 术语统一：主闭环=主流程。

### 5.1 主流程
- 起点：运营人员进入后台“表格导入”页面，看到初始学校文件队列或选择新的学校表格文件。
- 关键动作：选择/拖拽文件，查看统计卡片和文件队列，点击“全部开始”或行级“开始导入”，观察每个文件状态和进度变化。
- 输出结果：文件状态变为“成功”或“异常”；导入结果显示成功行数和问题数量；问题数据可按文件或批量导出 CSV。
- 管理反馈或决策回流：运营人员切换到“导入历史”查看已处理文件，针对异常文件导出问题数据给人工修正。

### 5.2 可补充展示的次级闭环
- 单文件导入闭环：在导入队列中选择某一行，点击“开始导入”，只处理当前文件。
- 异常处理闭环：异常文件产生问题数据，点击“导出问题数据”生成 CSV。
- 队列管理闭环：删除不需要的文件，同时删除该文件关联的问题数据。
- 历史查看闭环：导入成功或异常后的文件进入“导入历史”，可查看操作时间和结果。

## 7. 原型表达重点
- 移动端表达重点（如适用）：不适用，本项目端类型为 admin。
- 后台管理端表达重点：标准后台壳、清晰导航、卡片化内容区、数据表格、批量操作、行级操作、状态反馈和表格内部滚动。
- 首页/工作台重点：不单独做首页；“表格导入”作为默认落地页。
- 列表/表格页重点：导入队列表格必须清楚呈现学校文件、状态、进度、导入结果、说明和操作。
- 表单页重点：不做独立表单页；文件选择区域承担上传入口。
- 详情页重点：不做独立详情页；历史页展示处理后的摘要结果。
- 统计页重点：顶部 4 个统计卡片体现文件数量、总记录数、已入库、问题数据。

## 8. 风险点与待确认项
- 风险/待确认项：用户未指定项目名。
  - 影响：项目目录命名需要先稳定，否则后续 02/03/04/05 可能写入不同目录。
  - 建议处理方式：本次锁定为 `yunlong-data-import-admin`，后续如需改名必须回到 PROJECT 与 01 同步修改。

- 风险/待确认项：用户未指定具体视觉栈或组件库。
  - 影响：05 实现时可能在 Element Plus、原生 React/Tailwind 或自定义组件之间摇摆。
  - 建议处理方式：02/03/04 阶段继续使用 admin-ui 规则确定视觉和技术口径；04 再锁定实现方式。

- 风险/待确认项：状态只展示“待导入、成功、异常”，但模拟过程中需要内部进度阶段。
  - 影响：实现时可能把排队、校验中、导入中都暴露成独立状态，违背用户要求。
  - 建议处理方式：02 阶段明确“展示状态”只有三类，内部进度只通过进度条和说明字段表达。

- 风险/待确认项：页面整体不要滚动与内容较多存在冲突。
  - 影响：如果 02/03/04 不定义高度策略，05 容易出现整页滚动或表格溢出。
  - 建议处理方式：02 阶段必须写明主内容高度、左右布局高度、表格内部滚动策略。

## 9. 结论
- 本次原型建议优先围绕“学校文件批量导入队列”展开。
- 项目端类型已锁定为 admin，下一阶段应按 admin-ui 规则集进入《页面规格文件》编写。
