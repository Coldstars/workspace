# 高保真原型实现

## 阶段目标
实现可运行、可点击、可截图验收的高保真原型。

## 输入与输出
输入：`PROJECT.md`、`01`、`02`、`03`、`04`、`app/`。
规则：`workflow/阶段门禁与文档新鲜度规则.md`、`workflow/多页面项目规格组织规则.md`，按端类型读取 `mobile-ui/README.md` 或 `admin-ui/README.md`。
输出：`projects/<project-name>/app/`（执行清单按 `template.md`）。

## 阶段动作
1. 完成 app 门禁与影响判断，确认端类型和规则集匹配。
2. 建立 02 区域边界与 03 视觉检查项对照表。
3. **字段精确性核查（Must，实现每页前执行）**：对照 `02-页面规格文件.md` 对应页面的「页面结构」逐节核查，确认实现中每个卡片/列表/摘要区/表格列的字段均有明确来源；禁止自行增加 02 中未定义的字段，禁止自行删除 02 中标注「不可省略」的字段。
   - 问题案例：02 规格只列出「姓名、班级、加分/扣分」，实现时自行增加「学号、学年、状态说明、教师评语」等未定义字段，造成信息过载且与规格不符。
   - 预期收益：实现与规格精确对齐，消除 Agent 擅自发挥导致的字段漂移。
   - 影响范围：全部页面的字段实现，含后台表格列、筛选项、批量操作和行级操作。
   - 验证方法：截图检查 + 对照 02 文档逐字段核对；发现未定义字段必须移除或标注为「用户授权例外」。
   - 回滚条件：用户明确指示增加某字段时，需同步更新 02 规格文件后方可实现。
4. 先稳定基础层（token/页面壳/导航/通用组件/mock）。后台项目必须先搭 AppShell、Sidebar、Topbar、主内容容器，再实现具体页面。
5. 先主流程，后支撑页与边缘状态。
6. 检查可见可点元素都有真实行为。
7. 后台页面必须额外检查：侧边栏 active、面包屑、页面主操作、表格列、筛选、批量操作、行级操作、导出、异常处理、内部滚动。
8. 每完成一个主页面后必须截图验收，按匹配端类型的 `reference/12` 与 4 维度评分机制执行。
9. 完成全量截图、交互、可达性、构建回归。

## 阶段质量线
Must：未过门禁不改 app；不新增 02 外区域；每页实现前完成字段精确性核查；字段、表格列、操作必须与 02 一一对应；主流程与关键交互可演示；输出可验证结果；截图验收不达标不得继续下一页面。  
Should：重复结构优先组件化；记录与上版差异。  
Style：基础层 -> 主流程 -> 支撑页 -> 回归收尾。
