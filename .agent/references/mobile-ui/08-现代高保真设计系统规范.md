# 现代高保真设计系统规范

## 1. 目标
本文件用于在 03 视觉规范阶段产出可执行的项目级设计系统，避免只写“简洁、现代、高级”等空泛风格词。

设计系统必须能约束：
- 页面结构模式
- 色彩体系
- 字体层级
- 间距和密度
- 组件形态
- 图标和状态
- 动效反馈
- 反模式和交付检查

## 2. 设计系统输出结构
视觉规范必须输出以下字段：

- Pattern：页面结构模式，例如信息概览、任务处理、检查表单、对象详情、统计分析。
- Style：3-5 个风格关键词，每个关键词必须能对应界面规则。
- Colors：主色、辅助色、状态色、背景色、文字色。
- Typography：字体族、字号层级、字重数量、数字和单位规则。
- Spacing：间距系统、页面密度、模块内外距离。
- Components：卡片、列表、表单、按钮、导航、标签、图表、弹窗的表现。
- Effects：圆角、阴影、边框、过渡、按压、选中、提交、加载、错误反馈。
- Avoid：本项目明确不要出现的视觉问题。
- Checklist：交付前 UI 检查项。

## 3. 色彩规则
- 默认使用 `60/30/10` 色彩比例：
  - 60%：页面背景、基础容器、中性色。
  - 30%：卡片、模块、弱强调区域。
  - 10%：主按钮、关键数据、告警、选中态。
- 主色只用于关键路径，不要把所有卡片、图标和按钮都染成同一色。
- 状态色必须语义化：成功、警告、风险、处理中、中性分别固定。
- 禁止大面积随机渐变、无业务含义的霓虹色、同屏过多刺眼原色。
- 禁止为了“高级感”随意使用跨远色相渐变，例如蓝绿、紫橙、红蓝等没有明确业务语义的拼接。
- 渐变优先用于 Hero、摘要卡、数据汇总卡等重点区域；应使用项目声明的专业品牌色、同色系或邻近色，并保证文字对比度。
- 如确需使用互补色或跨色相组合，必须在项目视觉规范中说明业务语义，并限制在状态、图表或局部提示中，不能作为主视觉底色。
- **背景-容器-卡片层级可辨（Must）**：页面背景、模块容器、卡片内容必须能被肉眼区分，但不强制使用灰色页面背景或卡片阴影。
  - 允许页面背景使用纯白色，适用于阅读型、内容型、极简型或品牌留白型页面。
  - 允许卡片无阴影，前提是通过描边、分割线、间距、分组标题、背景色差、色块、排版密度或状态标识形成可辨层级。
  - 白色背景上可以出现白色卡片，但必须有描边、阴影、分割线、间距、分组标题或其他明确层级手段。
  - 模块内容区（如 Hero 区、分组标题区）不得与周边内容混成无法辨认的一整块平面。
  - 问题案例：白色页面背景上堆叠白色卡片，且无描边、无阴影、无分组标题、无足够间距，用户无法判断卡片边界。
  - 预期收益：允许更灵活的极简/内容型视觉，同时确保信息分组清晰、卡片边界可辨。
  - 影响范围：全部页面的卡片和容器实现。
  - 验证方法：截图后检查页面背景、模块容器和卡片边界是否可被肉眼识别；纯白背景和无阴影卡片不自动扣分。
  - 回滚条件：若项目采用实验性无边界排版，必须在 03 视觉规范中说明业务语义，并在关键交互区域保留可识别的点击边界。

## 4. 间距与扫读
- 默认遵循 8 点间距系统，核心间距从 `4 / 8 / 12 / 16 / 20 / 24 / 32` 中选择。
- 模块之间的距离必须大于模块内部元素距离。
- 信息组织遵循 F-pattern：顶部先给页面身份和关键结论，再给核心模块，再给细节列表。
- Thumb Zone：主操作优先放在屏幕底部 1/3 或首屏明确操作区。
- 首屏必须出现主价值信息，不能只出现大标题和空白装饰。

## 5. 字体层级
- 单项目最多使用 4 个字号、2 个字重。
- 建议层级：
  - 页面标题：`18-20px`，`600/700`
  - 模块标题：`15-17px`，`600`
  - 正文：`14-15px`，`400/500`
  - 辅助信息：`12-13px`，`400`
  - 数据强调：可放大，但必须服务指标识别。
  - **核心指标数字（Must）**：工作台/统计页的核心业务指标数字（如次数、分值、人数）字号不得低于 `24px`，字重不得低于 `600`；同屏规则保持一致。
    - 问题案例：Hero 区「今日检查 12 次」的数字「12」与正文等大，首屏看不出重点指标。
    - 预期收益：指标数字冲击感强，用户一眼识别关键数据。
    - 影响范围：所有含指标卡的首页、统计页、详情摘要区。
    - 验证方法：截图检查 D3 业务语义维度，指标数字是否显著大于正文。
    - 回滚条件：纯文字页（如表单、说明页）无指标卡不受此约束。
- 数字、金额、比例、排名等可使用更强字重，但同屏规则保持一致。

## 6. 高保真细节
- 卡片必须有明确层级：可通过背景、边框、轻阴影、间距、分组标题、色块、状态标识等方式区分，标题、正文、状态、操作不能混成一团。
- 列表不能只是同质白卡堆叠，必须通过状态、图标、标签、分组标题或时间线强化识别。
- 成功态、空态、错误态需要有情绪反馈：图标/插图、简短文案、下一步操作。
- 图片只在能增加真实感时使用，禁止随机库存图撑场面。
- 组件库默认视觉不得替代项目设计系统。

## 7. 微交互
- 常规过渡使用 `150-300ms`。
- 按钮、卡片、列表项至少具备 press/active 反馈。
- Toast、弹窗、提交成功、错误提示必须有出现和消失反馈。
- 动效只用于解释状态变化，不做无意义炫技。

## 8. 反模式
- 随机渐变、随机阴影、随机毛玻璃。
- 无业务语义的跨远色相渐变，例如蓝绿、紫橙、红蓝大面积拼接。
- **大面积主色卡片允许，但原色/刺眼纯色直铺 Must 禁止**：Hero 卡/摘要卡/数据汇总卡可以使用大面积主色、深品牌色或同色系渐变作为卡片底色，例如 `#0052D9` 这类专业深蓝可用于整卡背景；但不得把原色、满刺激色或刺眼霓虹色直接铺满整卡，例如 `#FF0000`、`#0000FF`、`#00FF00`、`#FFFF00`、`#FF00FF`、`#00FFFF` 及接近这些原色的高刺激颜色。
  - 问题案例：用 `#FF0000` 作为整张风险 Hero 卡背景，视觉刺眼、可读性差，且把普通风险表达成紧急警报。
  - 预期收益：保留品牌主视觉冲击力，同时避免原色直铺造成廉价感、压迫感和阅读负担。
  - 影响范围：全部 Hero 卡、摘要卡、数据汇总卡、角色首页主视觉卡等大面积色块。
  - 验证方法：截图检查大面积色块是否来自项目声明的品牌 token，文字/图标对比是否充足，是否避免原色或霓虹色直铺。
  - 回滚条件：极端告警、安全事故等强警示场景可短时使用高刺激色，但必须在项目视觉规范中说明业务语义，并限制在告警页或局部告警条，不作为常规卡片底色。
  - 正确做法：`primaryDeep: #0052D9` 整卡底色 + 白色文字 + 低透明度同色系渐变；或深青/深绿等专业品牌色主视觉卡。
  - 错误做法：整张卡片 `background: #FF0000`、`background: #0000FF`，或红蓝高刺激拼接渐变。
- **主色不统一（Must 禁止）**：主按钮、关键数字标签、图表主数据线、选中态、Hero 区品牌元素必须使用同一主色 token，不得出现一个页面内多套蓝色并存（如按钮 #1677FF、图表 #8884d8、图标 #3B82F6 各用各的）。
  - 问题案例：recharts 默认色 + Tailwind 蓝 + 自定义渐变蓝三套蓝色同屏，视觉凌乱。
  - 预期收益：品牌感统一，专业度提升。
  - 影响范围：全部含图表、按钮、状态标签的页面。
  - 验证方法：截图检查主色 hex 是否唯一且与 03 视觉规范 token 一致。
  - 回滚条件：图表多系列数据线允许使用语义色组，但主数据系列必须使用主色。
- 所有卡片长得一样，没有主次、状态或层级边界。
- 一屏只有文字，没有图标、标签、数字或视觉锚点。
- 空态、成功态、异常态只有一行文字（Must 禁止：必须有图标 + 文案 + 下一步操作）。
- 重要操作放在拇指难触达区域，底部按钮遮挡内容。
- 只写风格关键词，没有落到颜色、字号、间距、组件和状态。

## 9. 交付前检查
- 首屏是否能一眼看出页面目标和主任务。
- 主色是否只服务关键动作和重点信息。
- **大面积主色卡片是否合规**：允许专业深主色或同色系渐变，例如 `#0052D9`；禁止 `#FF0000`、`#0000FF` 等原色/刺眼纯色整卡直铺。
- **页面背景、容器、卡片层级是否可辨**：允许纯白背景和无阴影卡片，但必须通过描边、间距、分组标题、色块或排版层级区分边界，禁止同色无边界平铺。
- **主色是否统一**：按钮、图表主线、重点标签、选中态的颜色 hex 是否均来自同一 token。
- **核心指标数字是否 ≥ 24px / 600 字重**，与正文有明显视觉差。
- **页面顶部是否直接从顶部导航栏开始**，无任何模拟手机状态栏元素。
- **有底部固定操作栏的页面**，内容区底部是否预留了足够 padding，最后一条内容是否可见。
- 字号、字重、圆角、阴影是否稳定。
- 图标是否统一且辅助理解。
- 主要状态是否有图标、颜色、文案和下一步动作。
- 空态、成功态是否有图标 + 文案 + 下一步操作（不接受纯文字）。
- 页面在默认移动端验收视口下是否仍有清晰层级。
- Pattern、Style、Colors、Typography、Effects、Avoid、Checklist 是否可执行。
