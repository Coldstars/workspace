# React 高保真实现规范

## 1. 定位
本文件只服务阶段 5 `app/` 实现，聚焦 React + Vite + Tailwind 的代码结构、视觉还原、性能和截图检查。不定义门禁，不改变 01-04 文档。

## 2. 实现顺序
1. 从 03/04 抽取设计 token、组件规则、主链路和验收视口。
2. 先稳定 `tailwind.config.js`、`src/styles.css`、基础 UI 组件和 mock 数据。
3. 再按页面纵切实现：布局、数据、交互、状态、移动端适配一起完成。
4. 每完成一个主页面就检查目标视口，发现风格漂移先修 token/组件。

## 3. 推荐结构
```text
src/
  App.tsx
  main.tsx
  styles.css
  types.ts
  data/
  components/
    ui/
    charts/
    layout/
  pages/
```

要求：
- `App.tsx` 只放顶层状态和路由，不堆页面细节。
- 页面负责组合，基础视觉进入 `components/ui/`。
- mock 数据集中在 `data/`，不要散落在 JSX。
- 巨型文件按重复结构小步拆分，不一次性无依据重写。

## 4. 高保真还原
- 颜色来自 Tailwind token 或语义 class，不随机写 hex。
- 页面壳、卡片、按钮、标签、空态、图表容器优先组件化。
- 图标统一用 `lucide-react`，尺寸和风格稳定。
- 列表页需要状态、标签、分组、图标或时间线，避免同质白卡。
- 表单页必须有输入、校验、提交反馈和恢复路径。
- 统计页必须处理图表容器高度、Tooltip、Legend、空态和加载态。
- 通用 `Card`、`Button`、`MetricCard` 等组件应通过 `variant`、`tone` 或等价属性控制背景与文字对比，禁止页面层依赖 class 拼接覆盖默认背景/文字色导致白底白字。
- **图表主色必须使用 token，禁止使用 recharts/组件库默认色**：柱状图、折线图、面积图的主数据系列颜色必须显式设置为项目主色 token（如 `fill={colors.primary}`），不得依赖 recharts 默认的 `#8884d8` 等颜色。
  - 问题案例：recharts 默认蓝紫色 + Tailwind 主色按钮 + Hero 渐变三套蓝同屏并存，品牌色凌乱。
  - 预期收益：图表与按钮、标签颜色统一，专业感显著提升。
  - 影响范围：所有使用 recharts 的图表页面（P05/P07/P13 等）。
  - 验证方法：截图对比图表主色与按钮颜色是否视觉一致；代码审查图表 `fill`/`stroke` 属性是否使用 token 变量。
  - 回滚条件：多系列对比图表可使用语义色组，但主系列必须是主色 token。

## 5. React 实践
高优先级：
- 避免巨型组件和散落样式。
- 重图表、地图、上传、编辑器等非首屏能力可懒加载。
- 图片、图表、异步内容必须有尺寸、宽高比或 skeleton，避免布局跳动。
- 动画只用 `transform` 和 `opacity`。

中优先级：
- 搜索、滚动、resize 等高频交互使用 debounce/throttle。
- 列表使用稳定 key，不用可变列表 index。
- 派生数据确实重复且影响渲染时再 `useMemo`。
- 长列表超过 50 条时用分页、分组、折叠、虚拟列表或限制渲染数量。

低优先级：
- 小函数缓存、数组方法微优化只在明显卡顿时处理。
- 不为简单组件过度 memo。

## 6. 交互与无障碍
- 主按钮、图标按钮、列表点击热区不低于 44px。
- 可点击控件具备 press/active/focus-visible。
- 图标按钮有 `aria-label` 或可见文字。
- 可见文字按钮或图标按钮必须绑定真实处理函数；不能只显示按钮样式或可点击文案。
- 表单输入有可见 label，错误提示靠近字段。
- 关键操作不依赖 hover-only。
- 弹窗和抽屉必须可关闭，危险操作需要二次确认或撤销。

## 7. 截图验收硬门槛
阶段 5 的交付对象是可用于客户演示的高保真交互效果图，不是功能 demo。构建通过、页面可达、主流程可点击都不能替代截图验收。

最低截图覆盖类型：
- `375x812`：入口/角色选择/首个落地页；若项目没有入口页，则选择默认首页。
- `375x812`：每个主要角色的主页面/工作台/首页，至少覆盖 2 个角色；若项目只有 1 个角色，则覆盖该角色主页面。
- `375x812`：核心表单、主操作或核心任务流页面。
- `390x844`：核心详情页或结果页。
- `390x844`：统计、图表、看板或数据密度最高页面；若项目无图表页，则选择列表密度最高页面。

截图页选择规则：
- 截图页必须从 `02-页面规格文件.md` 的页面清单和主演示流程中选择。
- 多角色项目必须优先覆盖角色入口和各主要角色首页，避免只验收单一角色。
- 若一个页面同时满足多个覆盖类型，可以复用，但最低仍不得少于 4 张截图；多角色项目不得少于 5 张。
- 最终汇报必须列出实际截图页面 ID、页面名称、视口、覆盖类型、截图状态、D1/D2/D3/D4 和结论。

截图覆盖完整性（Must）：
- 阶段 5 若环境可截图，必须完成项目 `04-开发标准和验收标准.md` 中声明的全部核心截图计划；若项目 `04` 未声明截图计划，则按本节“最低截图覆盖类型”生成截图覆盖清单。
- 发现第一张或中途某张截图不达标时，可以先修复该页；该页复测达标后必须继续完成剩余截图，禁止因为某一页修复中断整个验收。
- 未完成全部截图覆盖时，最终回复必须写明“截图验收未完成”，禁止使用“验收通过”“交付完成”“视觉达标”等结论性表述。
- 每次阶段 5 最终汇报必须包含截图覆盖矩阵，字段为：页面 ID、页面名称、视口、覆盖类型、截图状态、D1、D2、D3、D4、总分、结论。
- 问题案例：只截 P00 入口页并发现问题，修复 P00 后直接结束，未继续检查 `04` 中要求的 P01/P02/P09/P13/P17，导致项目级视觉验收被误判为完成。
- 预期收益：单页修复不会吞掉剩余验收任务，项目级交付状态可被用户和后续 agent 快速核对。
- 影响范围：所有阶段 5 app 实现、UI 调优、截图验收、交付汇报任务。
- 验证方法：最终回复检查是否包含完整截图覆盖矩阵；矩阵必截页面不得缺失，且每张截图必须有四维评分和结论。
- 回滚条件：仅在用户明确要求“只检查某一个页面/只做局部截图”时可不执行项目级覆盖，但必须说明使用显式例外，并禁止宣称项目级验收通过。

截图必须检查：
- 首屏是否有明确主视觉、真实业务数据密度、页面目标和主操作。
- 页面是否像移动端高保真效果图，而不是线框图、普通表单页、后台列表页或组件库默认页。
- **页面顶部是否直接从顶部导航栏开始，无任何模拟手机状态栏（信号/电量/时间）**。
- **页面背景、模块容器、卡片层级是否可辨**：允许纯白背景和无阴影卡片，但必须通过描边、间距、分组标题、色块、分割线或排版密度区分边界；禁止同色无边界平铺。
- **大面积主色卡片是否合规**：Hero 卡、摘要卡、数据汇总卡可以使用专业深主色或同色系渐变，例如 `#0052D9`；不得使用 `#FF0000`、`#0000FF`、`#00FF00` 等原色/刺眼纯色整卡直铺，且文字对比必须充足。
- **主色是否统一**：截图中按钮、图表主数据线、重点数字/标签的颜色是否均来自同一品牌色 token，无多套蓝色/品牌色并存。
- **核心指标数字是否 ≥ 24px / 600 字重**，与周围正文有明显视觉层级差异。
- **底部导航/底部操作栏是否遮挡内容（一票否决项）**：凡截图显示最后一条列表项、表单字段或操作按钮被底部固定区遮挡，D4 强制判定 ≤ 1 分，必须进入修复循环。
- **页面字段是否与 02 规格文件一致**：实现的卡片/列表字段必须能在 02 对应页面结构中找到明确来源，不允许自行增减字段（见字段精确性规则）。
- 文本、标签、按钮、卡片、图表是否溢出、拥挤、错位或层级混乱。
- 图表是否具备可读的标题、单位、Tooltip、Legend 和稳定容器尺寸；图表主色是否使用项目 token 而非 recharts 默认色。
- Toast、Dialog、Popup、筛选抽屉等浮层层级是否正确，是否遮挡主操作或被裁切。
- 视觉 token 是否真实生效，例如页面背景、主色、状态色、卡片阴影、圆角、按钮样式和图表颜色。
- 多角色项目必须检查各主要角色首页的首屏视觉和关键交互，不能只验收单一角色首页。

风格漂移处理：
- 如果截图显示页面像线框图、组件堆叠、普通 demo、默认组件库页面或低保真灰白稿，必须先修视觉 token、基础 UI 组件、页面壳和核心页面母版，再继续铺其他页面。
- 如果截图显示核心页面首屏不达标，不得用“全页面已实现”“主流程已跑通”“构建通过”作为交付理由。
- 如果 Tailwind token、PostCSS、组件库皮肤或依赖版本导致视觉不生效，必须先修配置和基础层，再修页面细节。

截图工具规则：
- 如环境已有浏览器、Playwright、Puppeteer 或等价工具，必须使用可用工具完成截图检查。
- 如环境没有截图依赖，不自动新增依赖，不擅自安装浏览器。
- 无法截图时，最终汇报必须明确说明原因，并执行替代检查：`npm run build` 或等价构建、目标视口审查、DOM/CSS token 检查、关键页面人工预览说明。
- 无法截图不代表验收通过，只能标记为“截图硬门槛未完全覆盖”，并说明残留风险。

## 7.5 Agent 自动化视觉质量评分（4 维度，强制执行）

本节是阶段 5 每张截图后的强制自评机制。每个维度提供 0-5 分锚点描述，防止 Agent 自评虚高。

### 评分维度（每项 0-5 分，满分 20 分）

**维度一：结构与层级 (Structure & Hierarchy)**
- **5 分**：页面有明确主视觉区（Hero/统计卡）、功能区（操作/列表）、辅助区三层，层级通过阴影、背景色差、描边、间距、分组标题或色块清晰建立，卡片有主次
- **4 分**：主次分层存在，纯白背景或无阴影卡片仍能通过描边、间距、分组标题等方式被识别，局部略扁平但整体结构可识别
- **3 分**：有分组标题或少量图标/颜色区分，但局部卡片边界偏弱，扫读时需要停顿
- **2 分**：全页同质白卡或裸列表，缺少阴影、描边、分组、间距等层级手段，无主次，像纯列表页
- **1 分**：纯文本堆叠，无任何结构化卡片或分层处理
- **0 分**：布局崩溃，内容重叠，无法阅读

**维度二：质感与风格 (Fidelity & Style)**
- **5 分**：主色品牌色真实生效，圆角/阴影/背景色差符合项目设计系统，大面积专业主色卡片（如 `#0052D9`）对比充足且整体像高保真移动端效果图
- **4 分**：主色生效，圆角阴影基本到位，局部细节略有缺失；大面积主色卡片不刺眼且文字可读
- **3 分**：有主色但应用范围不足（仅 1 个按钮）；或圆角阴影存在但与设计系统出入较大；或大面积主色卡片质感一般但未使用原色直铺
- **2 分**：主色缺失，完全使用组件库默认颜色，像普通 CRUD 页；或整卡使用 `#FF0000`、`#0000FF` 等原色/刺眼纯色导致观感廉价
- **1 分**：黑白灰线框图，无圆角阴影，无品牌色；或大面积色块文字对比严重不足
- **0 分**：CSS 崩溃，完全无样式

**维度三：业务语义与数据表达 (Semantic & Data)**
- **5 分**：核心业务数字（分数/金额/排名）字号大字重强；成功/警告/危险状态均有语义色 + 独立图标；数据与说明文字层级清晰可辨
- **4 分**：关键数据有强调，状态颜色和图标覆盖主要场景
- **3 分**：核心数据有些强调，状态颜色或图标有遗漏
- **2 分**：数据字号字重无区分；状态只靠颜色或只靠文字，无图标
- **1 分**：业务数据与说明文字完全无区分，状态无语义色
- **0 分**：无业务数据呈现，无状态区分

**维度四：排版与边界容错 (Typography & Layout)**
- **5 分**：文字无溢出，行高/间距舒适遵循 8pt 系统，底部操作栏已预留 padding 不遮挡内容，Flex 对齐整洁
- **4 分**：整体对齐良好，个别元素间距略紧，底部操作栏安全
- **3 分**：局部文字拥挤或单行截断，底部操作栏无遮挡问题
- **2 分**：文字溢出容器，有明显对齐错位（无底部遮挡）
- **1 分**：多处布局错误，文字大量溢出
- **0 分**：整体布局崩溃
- **⚠️ 一票否决（底部遮挡）**：截图显示任何列表末行、表单末项、正文内容被底部固定操作栏/导航栏遮挡 → D4 强制判定为 **1 分**，无论其他排版是否良好，总分必然 < 16 分，强制进入修复循环。
  - 问题案例：页面列表最后一条记录被底部 TabBar 覆盖，用户滚动到底也无法看到完整内容。
  - 修复方式：为内容容器添加 `pb-[80px]` 或 `pb-safe`（含 safe-area-inset-bottom），并验证最后一条内容在截图中完整可见。
  - 回滚条件：无底部固定元素的页面不受此规则约束。

### 判定逻辑

| 条件 | 判定结果 |
|---|---|
| 每个维度 ≥ 4 分，且总分 ≥ 16 分 | **达标**，可继续下一页面或交付 |
| 任一维度 < 4 分，或总分 < 16 分 | **不达标**，强制进入修复循环 |
| 截图不可用 | 跳过打分，执行 7.4 替代检查，标记"视觉质量评分未完全覆盖" |

覆盖完整性判定：
- 单张截图达标只代表该页面达标，不代表项目级截图验收通过。
- 项目级截图验收通过条件：截图覆盖矩阵中所有必截页面均已截图，且每张截图每个维度 ≥ 4 分、总分 ≥ 16 分。
- 截图不可用时只能执行替代检查，并标记“截图硬门槛未完全覆盖”；不得视为视觉验收通过。

### 强制动作（不达标时必须按序执行，不得跳过）

1. **输出《UI 调优执行清单》**：每条格式为：`[维度] → [具体问题] → [修改文件/组件] → [预期改后效果]`
2. **修复代码**：优先修 token/基础组件，再修页面细节
3. **重新截图**并再次执行 4 维度打分，明确输出各维度分数（D1/D2/D3/D4）和总分
4. **最多 3 轮自修复循环**；超过 3 轮必须暂停，向用户说明卡点原因，禁止继续循环并以"修复中"状态交付
5. 达标后方可继续下一页面

### 反偏差规则（Must，与评分同步执行）

**规则 1：先找缺陷再打分**
- 打分前必须先输出"缺陷清单"，格式：`[维度] 缺陷描述`
- 每个维度**至少列出 1 个可改进点**（即使整体质量已高）
- 如果某维度缺陷清单为空，该维度最高只能给 4 分（完美需要证明）

**规则 2：锚点举证**
- 每个维度打分时必须引用对应的锚点描述，并说明当前截图符合哪条锚点
- 禁止只输出分数不输出锚点依据
- 示例：`D1=4：主次分层存在（Hero 统计卡 + 下方列表），但卡片间阴影差异不明显（符合锚点"局部元素略扁平"）`

**规则 3：对照 03 视觉规范举证**
- 至少引用 03 视觉规范中 2 个具体 token（如主色 hex、圆角值、字号）
- 说明截图中对应元素是否与 token 一致
- 如找不到对应 token 或 token 未生效，对应维度扣至少 1 分
