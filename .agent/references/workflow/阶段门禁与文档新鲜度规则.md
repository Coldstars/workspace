# 阶段门禁与文档新鲜度规则

## 1. 项目锁（Must）
- 当前项目必须由用户明确指定：`projects/<project-name>`。
- 涉及 `projects/` 但未指定项目名，先追问。
- `PROJECT.md`、`01/02/03/04`、`app/` 必须来自同一项目。
- 只改 `.agent` 时不绑定项目。

## 2. 端类型锁（Must）
- `PROJECT.md` 必须明确端类型：`mobile / admin / mixed`。
- 端类型缺失时，不进入 `02/03/04/05`。
- 端类型决定 reference 读取：
  - mobile -> `references/mobile-ui/README.md`
  - admin -> `references/admin-ui/README.md`
  - mixed -> 两套 README 均需读取，逐页按所属端执行。
- 端类型从 `mobile` 改为 `admin`、从 `admin` 改为 `mobile`、或改为 `mixed`，视为结构级变更，必须回到 `01` 并复核 `02/03/04`。

## 3. 文档有效性（Must）
文件“存在”不等于“有效”。有效需同时满足：
- 与当前需求一致。
- 与当前 reference 一致。
- 与项目端类型一致。
- 与上下游阶段一致。
- 位于当前项目目录。

## 4. 规则分层（唯一口径）
- Must：阻断项。
- Should：提示/扣分项。
- Style：风格建议。

默认：门禁、顺序、项目锁、端类型、有效性属于 Must。

## 5. 影响矩阵（Must）
- 端类型/页面形态/目标/角色/范围/主闭环变化 -> 更新 `01`，并复核 `02/03/04`。
- 页面/导航/入口/跳转/状态/布局变化 -> 更新 `02`。
- 表格列/筛选项/批量操作/行级操作/后台页面模块变化 -> 至少更新 `02` 与 `04`。
- 风格/token/图标/图表/动效/组件质感变化 -> 更新 `03`。
- 后台视觉体系/布局壳/表格密度/滚动策略变化 -> 至少更新 `03` 与 `04`。
- 技术栈/依赖/目录/验收口径变化 -> 更新 `04`。
- 可达性/交互行为/截图验收/构建稳定性变化 -> 进入 `05` 改 `app`。

处理顺序：`01 -> 02 -> 03 -> 04 -> 05`

## 6. app 门禁（Must）
改 `app/` 前必须确认：
- 项目锁稳定。
- 端类型已锁定。
- 已读取匹配端类型的 reference set。
- `01/02/03/04` 存在且有效。
- 四份文档之间及与 reference 无冲突。

未通过门禁：禁止改 app。

## 6.5 阶段 5 出口门禁（Must）
涉及 `app/` 的阶段 5 任务，若 `04-开发标准和验收标准.md`、`mobile-ui/12-React高保真实现规范.md` 或 `admin-ui/12-React后台实现规范.md` 要求截图验收，则截图覆盖矩阵是阶段 5 出口材料。

- 未输出截图覆盖矩阵，或覆盖矩阵缺少必截页面时，阶段 5 不得标记为完成。
- 单张截图达标只代表该页面达标，不代表项目级截图验收通过。
- 构建通过、页面可达、主流程可点击不能替代截图覆盖矩阵。
- 问题案例：只完成入口页截图并修复局部问题后，未继续执行 04 中声明的核心页面截图计划，却在汇报中暗示视觉验收已完成。
- 预期收益：防止阶段 5 因单页问题中断全量验收，确保交付状态真实可追溯。
- 影响范围：所有涉及 `app/` 的阶段 5 实现、UI 调优、截图验收和最终交付汇报。
- 验证方法：最终回复必须能找到截图覆盖矩阵，且矩阵包含项目 04 声明的全部核心截图计划或 12 的最低覆盖类型。
- 回滚条件：用户明确要求“只检查某一页/只做局部截图/不做完整验收”时，可作为显式例外跳过项目级覆盖，但必须声明截图验收未完成，禁止使用“验收通过”“交付完成”“视觉达标”等项目级结论。

## 7. 显式例外（Must）
用户明确说“只改 app / 只修 bug / 不改文档 / 只改 .agent”时可跳过文档更新，但需在回复里说明。

## 8. 持续治理
### 8.1 规则准入（Must）
每次加/改规则必须包含：问题案例、预期收益、影响范围、验证方法、回滚条件。

### 8.2 双周减法（Should）
每两周：删低命中、并重复、清冲突、统一表述。

### 8.3 波动追踪（Should）
建议记录：门禁通过率、视觉评分波动、规则回滚率、重复规则占比。

## 9. 执行前说明（Must）
需先说明：当前项目、端类型与规则集、受影响阶段、需更新文档、是否可进 app、是否使用显式例外。

## 10. 统一术语字典（Must）
文档与实现默认使用以下主术语；同义词仅用于映射，不作为主写法。

| 主术语 | 允许映射同义词 |
| --- | --- |
| 端类型 | 端、项目类型、页面形态 |
| 规则集 | reference set、规范库、读取地图 |
| 主流程 | 主链路、主演示流程、主闭环 |
| 页面可达性 | 可达、可进入、入口覆盖 |
| 状态反馈 | 页面状态、反馈态、状态表现 |
| 不可省略点 | 必须项、不得遗漏项、硬要求 |
| 视觉质量评分 | 视觉评分、UI评分、高保真评分 |
| Must/Should/Style | 硬约束/建议/风格建议 |

执行要求：
- 新增规则、模板和阶段文档优先使用主术语。
- 历史同义词可保留在引用或迁移说明中，但需在首次出现时映射到主术语。
- 同一文档内避免同一概念多种叫法。

## 11. 本次端类型规则补充说明
- 问题案例：后台管理页因默认读取移动端规范，生成缺少侧边栏、顶部栏、表格操作、批量处理和内部滚动策略的页面。
- 预期收益：让移动端与后台端各自走匹配规则，减少实现阶段反复返工。
- 影响范围：`PROJECT.md`、01/02/03/04、05 实现、视觉调优、截图验收。
- 验证方法：门禁表必须出现端类型与规则集；后台项目必须读取 `admin-ui/README.md`；02/03/04 必须包含后台专属字段。
- 回滚条件：若后续决定 workspace 只保留移动端生成能力，可移除 `admin-ui` 读取分支并恢复 mobile-only 门禁。
