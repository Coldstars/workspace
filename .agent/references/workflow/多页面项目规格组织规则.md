# 多页面项目规格组织规则

## 1. 适用范围
适用于页面数量较多、角色较多、流程较多、端类型较多，或页面信息结构差异明显的项目。

本规则不绑定具体行业、业务形态、端类型或页面名称。页面规格主文档必须优先保证用户可读、可改、可控。

## 2. 核心原则
- 页面多不是问题，用户看不懂才是问题。
- 多页面项目必须逐页清晰，而不是用复杂抽象压缩到不可编辑。
- 每个页面都应该像一张独立的信息架构稿。
- 页面结构按最终视觉分布从上到下罗列。
- 不同页面可以有不同的区域名称、区域数量和信息结构。
- 区域下面用短句罗列内容；操作、跳转、状态跟随对应区域书写。
- 页面模式、组件复用可以作为 agent 内部辅助，但不作为用户主阅读结构。
- 主流程、跳转和状态可以作为文档后部补充检查，不能替代逐页页面结构。
- 页面清单中的每个页面都必须有可达路径，不能只存在于清单中。
- 02 的逐页区域列表是 03/04/05 的结构边界；视觉规范和 app 实现必须逐页、逐区域对照 02，不得用模式化页面、视觉补偿或高保真增强新增 02 未定义结构。
- 02 改动后必须同步复核 03/04/app/mock/types：页面区域、字段、术语、状态或跳转一旦改动，下游文档、实现、mock 数据和类型定义不得残留旧口径。
- 被 02 删除的概念不得继续出现在视觉规范、验收标准、mock 数据、类型定义和 UI 文案中；除非用户明确要求保留页面名称、历史名称或兼容别名。

## 3. 逐页信息架构写法
每个页面至少包含：
- 页面名称。
- 所属端：`mobile / admin`。
- 页面用途。
- 页面入口与返回策略。
- 页面结构。
- 不可省略点。

页面结构写法：
- 按视觉顺序编号。
- 区域名称由当前页面自然生成。
- 区域下面直接写短句。
- 有字段就写字段。
- 有操作就写操作。
- 有跳转就写目标。
- 有状态就写状态表现。
- 有 Tab、筛选、弹窗、抽屉、步骤流，就写在对应区域下面。

## 4. 页面区域归纳方法
可从以下维度自然归纳区域，但不要强制所有页面一致：
- 视觉位置：顶部、首屏、滚动内容、固定底部、侧栏、浮层。
- 页面目标：浏览、录入、决策、处理、配置、汇报、协作、创作。
- 内容类型：摘要、列表、详情、表单、图表、媒体、地图、日历、消息、评论。
- 主操作：查看、筛选、搜索、提交、编辑、审批、拖拽、标注、对话、导出。
- 端类型：移动端、PC 后台、大屏、管理端、嵌入式容器。

## 5. 后台页面规格补充要求（Must）
后台项目的 02 页面规格必须逐页定义：
- 页面目标。
- 所属导航、侧边菜单位置、面包屑。
- 页面标题区、说明、主操作、次操作。
- 页面模块顺序：概览、筛选、操作区、表格、详情、问题处理、历史日志等按实际需要填写。
- 表格列、筛选项、批量操作、行级操作。
- 状态：空态、加载、成功、异常、部分成功、处理中、重试、导出可用/不可用。
- 弹窗/抽屉/Tab/步骤条出现位置与触发方式。
- 页面滚动与表格内部滚动策略。
- 页面可达路径、返回路径和跨页跳转。

后台页面不得出现：
- 只写“后台管理页/列表页/详情页”而不写具体模块。
- 只写“有筛选、有表格、有导出”而不写字段、操作、状态。
- 数据量大时只让整个页面滚动，不定义表格或内容区内部滚动。
- 在 03/04/05 阶段新增 02 未定义的菜单、卡片、字段、按钮或页面。

## 6. mixed 项目组织方式（Must）
mixed 项目必须拆成两类页面清单：
- 移动端页面清单：每页使用 `mobile-ui` 规则。
- 后台页面清单：每页使用 `admin-ui` 规则。

要求：
- 页面 ID 必须能区分端类型，例如 `M01`、`A01`。
- 两端可共享业务术语和 mock 数据，但不得共用同一套页面结构模板。
- 两端如存在同名业务对象，必须分别说明在移动端和后台端的呈现差异。

## 7. 页面多时怎么控制长度
- 不用模式继承替代逐页说明。
- 可以用短句，不写长段 PRD。
- 可以重复必要区域，保证用户能就地修改。
- 可以把通用实现提醒放在文档最后，但页面本身仍要独立可读。
- 可以把路由和状态汇总成表，但不能省略页面区域顺序。

## 8. 主流程与跳转最低要求
主流程与跳转必须覆盖：
- 主演示流程。
- 首页/入口到核心功能。
- 列表到详情。
- 详情到表单/处理。
- 表单提交后的结果或反馈。
- 弹窗、抽屉、Tab、步骤流等会改变页面状态的入口。
- 返回策略或回到哪里。

## 9. 页面可达性检查
页面清单中的每个页面都必须至少有一种进入方式：
- 入口页、导航栏、底部栏、侧栏或菜单进入。
- 首页/工作台/看板中的快捷入口进入。
- 列表项、卡片、记录行、消息项、图表排名项进入。
- 按钮、表单提交结果、弹窗操作或流程步骤进入。
- 特殊页面可通过演示角色切换、设置入口或调试入口进入，但必须在跳转说明中写清。

不可达页面必须处理：
- 如果页面确实不需要演示，从页面清单移除或标为不实现。
- 如果页面需要演示，补充入口、触发位置和返回策略。

## 10. 状态与反馈最低要求
状态与反馈必须覆盖当前项目会演示或影响还原的状态，例如：
- 默认态。
- 加载态。
- 空态。
- 筛选无结果。
- 编辑中。
- 校验失败。
- 提交成功。
- 提交失败。
- 处理中。
- 已完成。
- 权限/不可操作态。
- 后台批量操作中的部分成功、部分失败、可导出、不可导出、重试中。

状态名称不限于上述示例，必须从当前项目业务和交互中推导。

## 11. 坏例子
- 页面清单很完整，但逐页结构只有摘要。
- 前几页完整，后续页面只写“核心模块/操作/状态”。
- 每个页面强行套同一组区域名称。
- 每个区域强行拆成固定表格，导致用户改起来很累。
- 主文档写成页面模式、继承规则、差异规格。
- 为了减少重复，把页面结构抽象到用户看不懂。
- 页面清单里有页面，但主流程和补充跳转里找不到入口。
- 跳转关系散落，无法检查闭环。
- 状态只写“有空态/成功态”，不写出现位置和表现。
- 后台页面只写“列表 + 表单”，没有列、筛选、批量操作、行级操作和滚动策略。

## 12. 好例子
- 先列页面清单。
- 再逐页写页面用途和从上到下的页面结构。
- 每个页面区域名称贴近该页面真实内容。
- 操作、跳转和状态写在对应区域下面。
- 后台表格页明确筛选、列、主操作、批量操作、行级操作、空态和异常态。
- 每个页面都有不可省略点。
- 主流程、补充跳转和状态在文档后部集中查漏。
- 页面清单中的每个页面都能找到入口和返回方式。
- 实现 agent 可按页面区域顺序直接搭界面。
